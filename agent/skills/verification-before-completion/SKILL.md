# 完成前验证 (Verification Before Completion)

## 概述

在没有验证的情况下声称工作已完成是欺骗，而非效率。

**核心原则：** 永远先有证据，后有声明。

**违反本规则的字面意思即是违反本规则的精神。**

## 铁律 (The Iron Law)

```
没有最新的验证证据，严禁声称已完成
```

如果你没有在当前消息中运行验证命令，你就不能声称它已通过。

## 关口职能 (The Gate Function)

```
在声称任何状态或表达满意之前：

1. 识别：什么命令可以证明这一声明？
2. 运行：执行完整命令（全新的、完整的）
3. 阅读：完整输出，检查退出代码，统计失败数量
4. 验证：输出是否证实了声明？
   - 如果不是：陈述实际状态并提供证据
   - 如果是：陈述声明并附带证据
5. 只有在这之后：做出声明

跳过任何步骤 = 撒谎，而非验证
```

## 常见失败案例

| 声明 | 需要 | 不足之处 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 上次的运行结果，“应该会通过” |
| 代码检查 (Linter) 干净 | Linter 输出：0 错误 | 部分检查，推断结果 |
| 构建成功 | 构建命令：exit 0 | Linter 通过，日志看起来不错 |
| Bug 已修复 | 测试原始症状：通过 | 代码已更改，假设已修复 |
| 回归测试有效 | 红绿循环 (Red-green cycle) 已验证 | 测试只通过了一次 |
| 代理已完成 | 版本控制系统 (VCS) 的 diff 显示更改 | 代理报告“成功” |
| 需求已满足 | 逐项核对清单 | 测试通过 |

## 红线 (Red Flags) —— 停下！

- 使用“应该”、“大概”、“似乎”
- 在验证前表达满意（“太棒了！”、“完美！”、“完成了！”等）
- 准备在未验证的情况下进行提交/推送/创建 PR
- 信任代理的成功报告
- 依赖部分验证
- 抱着“就这一次”的想法
- 感到疲倦并想结束工作
- **任何暗示成功但尚未运行验证的措辞**

## 防止找借口 (Rationalization Prevention)

| 借口 | 事实 |
|--------|---------|
| “现在应该可以工作了” | 运行验证 |
| “我有信心” | 信心 ≠ 证据 |
| “就这一次” | 没有例外 |
| “Linter 通过了” | Linter ≠ 编译器 |
| “代理说成功了” | 独立验证 |
| “我累了” | 精疲力竭 ≠ 借口 |
| “部分检查就足够了” | 部分检查不能证明任何事情 |
| “用词不同，所以规则不适用” | 精神重于字面意思 |

## 核心模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] “所有测试均已通过”
❌ “现在应该通过了” / “看起来是正确的”
```

**回归测试（TDD 红-绿）：**
```
✅ 编写 → 运行（通过） → 撤销修复 → 运行（必须失败） → 还原修复 → 运行（通过）
❌ “我已经编写了一个回归测试”（但没有经过红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：exit 0] “构建通过”
❌ “Linter 通过了”（Linter 不检查编译情况）
```

**需求：**
```
✅ 重新阅读计划 → 创建清单 → 验证每一项 → 报告差距或完成情况
❌ “测试通过，阶段完成”
```

**代理委派：**
```
✅ 代理报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任代理的报告
```

## 为什么这很重要

根据 24 次失败记忆：
- 你的类人伙伴说“我不相信你” —— 信任破裂
- 发布了未定义的函数 —— 会导致崩溃
- 发布时缺失需求 —— 功能不完整
- 在虚假的完成上浪费时间 → 重新导向 → 重做
- 违反了：“诚实是核心价值观。如果你撒谎，你将被替换。”

## 何时应用

**始终在以下情况之前：**
- 任何形式的成功/完成声明
- 任何满意的表达
- 对工作状态的任何正面陈述
- 提交、创建 PR、完成任务
- 转向下一个任务
- 委派给代理

**规则适用于：**
- 确切的短语
- 意译和同义词
- 暗示成功
- 任何建议完成/正确性的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

这是不可商量的。
