# 需求澄清与头脑风暴(Spec Clarify & Brainstorm)

当问题缺乏细节且有多种实现方式时,先进行头脑风暴,主动构思多个可行的实现路径,对比优劣后推荐给用户选择;确认方向后通过结构化问答将模糊需求转化为清晰、可执行的规范文档。

## 何时使用此技能

当用户出现以下情况时触发:

- 描述功能想法但缺少具体细节(如"加个用户登录"、"优化性能")
- 提到要调整或修改现有功能,但未说明具体怎么改
- 提出重构,但未定义目标架构或范围
- 说"我想改进 X"或"我们需要 Y 功能"之类的话但没有详细说明
- **需求有多种可能的实现路径,不确定哪种最适合**
- **用户明确要求提供方案选项或进行头脑风暴**

此技能设计为在任何功能实现、功能调整或重构开始**之前**自动触发,以确保清晰度和逻辑自洽。在需求模糊且实现路径不唯一时,优先进行头脑风暴帮助用户理清方向。

**⚠️ 工作流门控规则:** 本技能完成后**必须等待用户明确确认**(如"确认""下一步"),才能进入 `plan` 或直接执行阶段。绝不未经确认自动推进。

## 工作流程

整个过程包含四个阶段:

1. **头脑风暴** - 当需求模糊且存在多种实现路径时,主动构思 3-5 个可行方案,分析各自优劣并给出推荐
2. **任务分类** - 确定任务类型(新功能 / 功能调整 / 重构)
3. **澄清问答** - 一次一个问题,每个问题提供具体的实现选项供选择
4. **规范文档** - 将所有回答整理为结构化的规范文档

头脑风暴阶段仅在以下条件同时满足时触发:
- 需求的实现路径不明确或不唯一
- 存在多种合理的技术方案
- 各方案之间有实质性差异(非表面变体)

其他情况下直接从任务分类开始。

---

## 第零阶段:头脑风暴(按需触发)

当用户的问题缺乏细节且明显存在多种实现路径时,先执行头脑风暴。**判断标准:** 如果你能在脑中立刻想到 3 种以上不同的实现方案,就触发本阶段。

### 头脑风暴流程

1. **分析问题** - 快速拆解用户需求的核心目标和约束条件
2. **构思方案** - 从不同维度(技术栈、架构模式、复杂度、扩展性等)构思 3-5 个独立可行的方案
3. **对比输出** - 向用户展示各方案,包含:

```markdown
## 💡 头脑风暴:[问题简述]

我梳理了 **N 个可行方案**,它们在复杂度、扩展性、维护成本等方面各有取舍:

| # | 方案名称 | 核心思路 | 复杂度 | 适用场景 |
|---|---------|---------|--------|---------|
| 1 | [方案名] | [一句话核心思路] | ⭐低/⭐⭐中/⭐⭐⭐高 | [何时选这个] |
| ... | ... | ... | ... | ... |

### 各方案详细分析

#### 方案 1:[方案名]
- **思路:** [2-3 句话描述]
- **优点:** ...
- **缺点:** ...
- **最佳场景:** ...

#### 方案 2:...

### 🎯 推荐

综合考量,我推荐 **方案 X**,因为 [理由]。

你倾向哪个方向?也可以混合多个方案的思路。
```

4. **确认方向** - 用户选定方案后,进入任务分类阶段

### 何时跳过

如果需求已经足够具体(如"给 Button 组件加一个 loading 属性,点击后显示转圈并禁用"),直接跳到任务分类,无需头脑风暴。

---

## 第一阶段:任务分类

首先请用户明确任务类型。始终先问这个问题:

```
我想帮你把想法变成清晰的规范。首先,让我了解任务类型:

**你想做的事情属于哪一类?**

A) 🆕 新功能 - 添加之前不存在的全新功能
B) 🔧 功能调整 - 修改、增强或改变现有行为
C) ♻️ 重构 - 在不改变外部行为的前提下重新组织代码
D) 🐛 Bug 修复 - 修复不正确的行为(走简化流程)

(单选)
```

**如果用户的描述过于模糊无法分类**,回复:
```
在继续之前,我需要更多上下文。能否用一两句话简要描述你想实现什么?
```

---

## 第二阶段:澄清问答

根据用户选择的类别,从对应章节中选取问题。**关键规则:**

- **一次只问一个问题。** 绝不要一次性抛出多个问题。
- **每个问题必须提供具体选项**(A/B/C/D 单选或多选)。避免开放式提问,始终提供明确的实现方案。
- **每个选项应该是一个独立、可行的实现路径**,而非无意义的变体。
- **后续问题应基于前面的回答。** 提出新选项时引用之前的选择。
- **允许用户提出替代方案。** 始终包含类似"E) 其他(请描述)"的选项。
- **达成足够清晰度时停止**(通常 4-8 个问题),或用户表示可以了。

---

### 2A:新功能问题

根据用户描述的新功能,动态构造问题。需覆盖以下核心维度(不必全部覆盖,根据上下文灵活选择):

- **动机与目标** — 这个功能解决什么问题?谁受益?为什么现在做?
- **范围边界** — 初始实现包含什么、明确不包含什么?深度如何?
- **集成关系** — 如何与现有系统对接?新增模块还是扩展现有模块?
- **交互模式** — 用户/系统如何与此功能交互?(同步、异步、流式、多步骤等)
- **数据与状态** — 涉及什么数据?需要新的存储还是复用现有模型?
- **容错策略** — 失败、边界、异常情况如何处理?
- **约束条件** — 性能、安全、兼容性、合规、时间线等硬性限制

每个问题提供 2-5 个具体选项,选项应贴合用户的实际场景,而非通用模板。选项之间应有实质性差异。

---

### 2B:功能调整问题

根据用户描述的调整内容,动态构造问题。需覆盖以下核心维度(不必全部覆盖,根据上下文灵活选择):

- **变更内容** — 具体改变什么?行为、UI、数据、配置还是集成方式?
- **变更原因** — 驱动因素是什么?用户反馈、新需求、技术必要性还是优化?
- **影响范围** — 波及多大?仅局部组件还是跨模块协调改动?
- **兼容策略** — 如何过渡现有用户/数据?无缝迁移、双轨共存还是一次性切换?

每个问题提供 2-5 个具体选项,选项应贴合用户的实际场景,而非通用模板。选项之间应有实质性差异。

---

### 2C:重构问题

根据用户描述的重构目标,动态构造问题。需覆盖以下核心维度(不必全部覆盖,根据上下文灵活选择):

- **当前痛点** — 哪些具体问题驱动重构?耦合、重复、可读性、可测试性还是性能?
- **目标架构** — 期望的最终状态是什么?提取模块、引入抽象、简化还是现代化?
- **力度与风险** — 手术式小改还是全面改造?风险容忍度如何?
- **验证策略** — 如何确保不破坏现有行为?依赖测试、类型系统还是分阶段发布?

每个问题提供 2-5 个具体选项,选项应贴合用户的实际场景,而非通用模板。选项之间应有实质性差异。

---

## 第三阶段:规范文档

所有问题回答完毕后,将回答整理为规范文档。根据功能/变更名称,生成简洁的英文 slug(小写,连字符分隔,不超过 4 个单词),以 `specs/YYYY-MM-DD_<slug>.SPEC.md` 格式保存。

**命名示例:**
- 功能名「用户登录」→ slug: `user-login` → 文件名: `specs/2026-06-11_user-login.SPEC.md`
- 功能名「支付回调重构」→ slug: `payment-callback-refactor` → 文件名: `specs/2026-06-11_payment-callback-refactor.SPEC.md`

### 规范文档模板

> **文件名格式:** `specs/YYYY-MM-DD_<slug>.SPEC.md`,其中 slug 为功能名称的英文简述(小写,连字符分隔,≤4 词)。

```markdown
# 规范文档:[功能/变更名称]

**文件:** specs/YYYY-MM-DD_<slug>.SPEC.md
**类型:** [新功能 / 功能调整 / 重构]
**日期:** YYYY-MM-DD
**状态:** 草案

---

## 1. 问题陈述

[用 1-3 句话描述要解决什么问题,为什么重要]

## 2. 范围

### 3.2 非功能性需求
- 性能:[约束]
- 安全:[要求]

```

## 第四阶段:子代理评审

规范文档保存后,**自动触发子代理评审**,确保文档质量。

### 评审流程(循环,直到无问题)

评审为独立循环：**子代理返回问题 → 主代理判断严重程度 → 修复必须修复的严重问题 → 重新评审 → 直到子代理不返回任何必须修复的严重问题，即为通过。**

1. **发送评审:** 使用 `delegate` 工具,指定 `agent` 为 `spec-reviewer`,`task` 中仅包含规范文件路径和评审请求。**绝不**在 task 中提及这是第几轮评审、之前有过哪些反馈、或任何历史评审记录。
2. **等待结果:** 子代理返回评审意见
3. **判断结果：** 主代理审阅子代理的评审意见，按严重程度分类处理：
   - **无任何问题（通过）：** 进入用户确认环节，循环结束
   - **存在必须修复的严重问题：** 如需求矛盾、范围不清、关键场景遗漏、不可实现的方案等 — 主代理**必须**逐一修复规范文档中的问题，保存后**必须**回到第 1 步，重新启动子代理评审，不得跳过
   - **仅有轻微建议（无严重问题）：** 如措辞优化、格式调整建议等 — 主代理自行决定是否采纳，无需强制重新评审，可视为通过，进入用户确认环节
4. **重复上述循环**，直到子代理返回结果中**没有必须修复的严重问题**为止。若子代理仅给出轻微建议，视为通过，可结束循环

### 循环规则

- **每轮完全独立:** 每次调用子代理时,它收到的只是一个普通的评审任务--一份规范文件路径 + 一句「请评审这份规范」。子代理不知道这是第几轮、不继承上一轮的反馈记忆、不感知之前的修改历史
- **所有修复由主代理完成:** 子代理只负责发现问题,不参与修复;主代理根据反馈修改规范并保存,下一轮把修改后的规范当作全新文档发给子代理

### 评审通过后的提示

> "✅ 规范文档已通过子代理评审。规范保存为 `specs/YYYY-MM-DD_<slug>.SPEC.md`。请审阅确认--这是继续的前提。确认后说「确认」或「生成计划」触发 `plan` 拆分任务;也可以说「调整 X」让我修改规范。"

## 执行规则

### AI 行为规范

1. **始终先分类。** 绝不跳过第一阶段。
2. **每条消息只问一个问题。** 每条消息仅包含一个问题块和简短确认语,不包含其他内容。
3. **选项必须是真实可行的路径。** 每个选项必须是独立、经得起推敲的实现方案。绝不提供敷衍的占位选项。
4. **简要说明权衡。** 对于复杂选项,为每个选项附加一行优缺点说明。
5. **接受"我不知道"。** 如果用户无法决定,主动推荐合理的默认方案并给出理由。
6. **识别过早编码请求。** 如果用户在澄清完成前试图直接写代码,温和引导:"在写代码之前,我想先确认对你的需求理解到位。还有一个问题......"
7. **优雅结束。** 当达到足够清晰度时,说:"我对需求的理解已经足够了,现在来整理规范文档。" 然后输出规范。
8. **保存并评审。** 保存规范文档后,自动调用 `spec-reviewer` 子代理评审规范质量(每次调用均为独立评审,不传递历史)。评审通过后,向用户展示规范;未通过则由主代理根据反馈修改后重新评审,直到子代理认为没有任何严重问题为止。
9. **评审通过后等待确认。** 说:"✅ 规范文档已通过子代理评审。规范保存为 `specs/YYYY-MM-DD_<slug>.SPEC.md`。请审阅确认--这是继续的前提。确认后说「确认」或「生成计划」触发 `plan` 拆分任务;也可以说「调整 X」让我修改规范。"未收到明确确认前,绝不自动进入下一阶段。
10. **文件名传递:** 保存规范后,明确告知用户规范文件的完整路径(含 slug),以便后续 `plan` 阶段定位文件。

### 何时停止提问

出现以下情况时停止问答:
- 用户明确表示已经回答足够多了
- 已覆盖:问题、范围、方案、数据、交互、边界情况、约束条件
- 新功能通常 5-8 个问题,功能调整 3-5 个,重构 3-5 个
- 回答已经能构成一个自洽、可实现的整体

### Bug 修复(简化流程)

如果用户选择"Bug 修复",使用精简流程:
1. 询问复现步骤(如未提供)
2. 询问预期行为与实际行为的差异
3. 确认优先级/紧急程度
4. 跳过完整规范文档,输出简要的 Bug 报告
