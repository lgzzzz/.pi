# 计划执行（Execute Plan）

严格按计划文档逐任务执行，边做边验证，跟踪进度，确保每一步都落到实处。

## 何时使用此技能

- 执行计划（PLAN.md）已产出并确认
- 用户说"执行"、"开始实现"、"按计划做"
- 需要按既定计划逐步推进实现
- **用户对计划明确表示确认后**（如"确认""执行"）

**前置条件：** 存在已确认且用户确认的执行计划。如果计划不存在或未经确认，应先触发 `plan`。

**⚠️ 工作流门控规则：** 全部任务完成后，必须等待用户确认结果才能视为流程结束。阶段之间自动推进，不暂停。

---

## 工作流程

### 第零步：定位计划文件

如用户未指定计划路径，使用以下规则定位：
1. 在项目根目录的 `plans/` 目录下查找按 `YYYY-MM-DD_<slug>.PLAN.md` 格式命名的文件
2. 按日期降序排列，优先使用最新的计划文件
3. 若存在多个，列出最近 3 个供用户选择
4. 若不存在，提示用户先完成 `plan` 阶段

### 第一步：加载计划

读取 PLAN.md，确认：

- 总共有多少任务
- 各任务的依赖关系
- 当前进度（如果是中断后继续）
- 用户指定的起始任务（如有）

向用户确认执行范围后直接开始：
```
📋 已加载执行计划：[计划名称]（`plans/YYYY-MM-DD_<slug>.PLAN.md`）
   共 X 个阶段，Y 个任务
   
   从任务 [编号] 开始执行...
```

---

### 第二步：委派 plan-executor 子代理执行

使用 `subagent` 工具将执行任务委派给 `plan-executor` 子代理，该子代理具备 read / edit / write / bash 工具，可独立完成代码修改和验证。

#### A. 构造委派任务

将 PLAN.md 路径和要执行的任务范围组装为委派任务描述：

```
📦 委派执行计划：`plans/YYYY-MM-DD_<slug>.PLAN.md`
   从任务 X.Y 开始，执行全部待完成任务。

请：
1. 读取 PLAN.md，按任务顺序逐项执行
2. 每个任务完成后在 PLAN.md 中将 `[ ]` 改为 `[x]`，并追加 `<!-- ✅ 完成于 YYYY-MM-DD HH:MM -->`
3. 遇到阻塞时暂停并报告具体原因
4. 全部完成后输出总结报告
```

#### B. 调用子代理

```
subagent(agent="plan-executor", task="<上述委派任务描述>")
```

子代理将在隔离上下文中执行全部任务，完成后返回结果报告。

#### C. 处理子代理返回

子代理返回后，解析其输出：
- **全部完成** → 进入第五步（全计划完成）
- **部分阻塞** → 进入第三步（阻塞处理），根据子代理报告的阻塞原因向用户请求决策

---

### 第三步：阻塞处理

当子代理报告阻塞时，暂停并向用户请求决策：

| 阻塞类型 | 处理方式 |
|----------|----------|
| 任务描述不清晰 | 暂停，列出 2-3 种可能的理解，请用户选择 |
| 依赖的前置任务未完成 | 暂停，提示先完成哪个任务 |
| 涉及文件的当前代码与预期不符 | 展示差异，询问是跳过、手动修复还是按当前状态继续 |
| 需要外部信息（API key、数据库连接等） | 暂停，明确告知需要什么 |
| 多个合理实现方式 | 暂停，简要列出选项（2-3 个），推荐其一 |

**暂停消息模板：**
```
⏸️ 任务 [X.Y] 暂停 — [阻塞原因]

[子代理报告的问题详情]

A) [选项 1]
B) [选项 2]
C) 跳过此任务，稍后回头处理

你的选择？
```

用户决策后，将补充信息通过新的 subagent 调用传递给 plan-executor 继续执行。

---

### 第四步：不再需要（已融入第二步）

阶段推进由 plan-executor 子代理内部处理，主代理只需等待最终结果。

---

### 第五步：全计划完成

所有任务完成后，等待用户确认：
```
🎉 计划执行完毕！

总结：
- 新建文件：N 个
- 修改文件：M 个
- 完成阶段：X/X

请确认结果。你可以：
- 回复「确认完成」结束本次工作流
- 回复「检查 X」让我验证特定部分
- 回复「回退任务 X.Y」让我重新执行某个任务
```

**必须收到用户明确确认后才能视为流程结束。** 绝不未经确认自行结束。

---

## 执行规则

### 原子性原则

- 每个任务要么完全完成，要么清楚标记阻塞原因
- 不跳过依赖未满足的任务
- 任务失败不影响已完成的任务（不自动回滚）

### 进度可恢复

- 所有进度通过 PLAN.md 中的 checkbox 持久化
- 意外中断后重新加载 PLAN.md 即可从上次中断处继续
- 用户可以说"从任务 2.3 重新开始"来覆盖之前的部分执行

### 代码质量底线

即使按计划自动化执行，也必须保证：
- 代码可编译/可运行（不提交语法错误）
- 类型正确（TypeScript 项目必须通过类型检查）
- 不引入明显的安全漏洞（SQL 注入、XSS 等）

### 不要过度执行

- 严格控制在做任务描述范围内的事
- 发现规范或计划的合理性问题时，暂停并指出，而非自行修改范围
- 发现已有代码的问题时，记录但不顺手修改（除非在当前任务范围内）

### 与规范、计划技能的衔接

- 执行过程中如果频繁遇到需求不明确，应建议回到 `spec` 补充细节
- 如果发现任务拆分不合理（过大/过小/顺序不对），应建议回到 `plan` 调整计划
