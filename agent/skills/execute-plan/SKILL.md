# 计划执行（Execute Plan）

严格按计划文档逐任务执行，边做边验证，跟踪进度，确保每一步都落到实处。

## 何时使用此技能

- 执行计划（PLAN.md）已产出并确认
- 用户说"执行"、"开始实现"、"按计划做"
- 需要按既定计划逐步推进实现
- **用户对计划明确表示批准后**（如"批准""确认""执行"）

**前置条件：** 存在已确认且用户批准的执行计划。如果计划不存在或未经批准，应先触发 `plan-from-spec`。

**⚠️ 工作流门控规则：** 全部任务完成后，必须等待用户确认结果才能视为流程结束。阶段之间自动推进，不暂停。

---

## 工作流程

### 第一步：加载计划

读取 PLAN.md，确认：

- 总共有多少任务
- 各任务的依赖关系
- 当前进度（如果是中断后继续）
- 用户指定的起始任务（如有）

向用户确认执行范围后直接开始：
```
📋 已加载执行计划：[计划名称]
   共 X 个阶段，Y 个任务
   
   从任务 [编号] 开始执行...
```

---

### 第二步：逐任务执行循环

对每个任务按以下子步骤执行：

#### A. 任务开始

输出当前任务信息：
```
🔨 正在执行 [任务 X.Y]：[任务名称]
   目标：[任务目标]
   涉及文件：[文件列表]
```

#### B. 读取上下文

在执行前读取所有**涉及文件**的当前内容。如果是新建文件，确认目标目录存在。

**关键原则：先读后写，避免覆盖已有改动。**

#### C. 实现变更

根据任务目标编写或修改代码。遵循：
- 只改任务范围内指定的内容，不做额外"顺便优化"
- 保持与现有代码风格一致
- 添加必要的注释（特别是非显而易见的逻辑）

#### D. 自检验证

对照任务完成标准逐项检查：
```
✅ 完成标准检查：
   [✅] 条件 1 — [验证结果]
   [✅] 条件 2 — [验证结果]
   [❌] 条件 3 — [未通过，因为...]
```

如果未通过，立即修复后重新检查。

#### E. 任务完成标记

任务通过所有检查后，在 PLAN.md 中标记为已完成：
```markdown
   - [x] 任务 X.Y 完成
```

**标记方式：** 在 PLAN.md 中找到对应任务的完成标准 checkbox，将 `[ ]` 改为 `[x]`，并在任务后追加 `<!-- ✅ 完成于 YYYY-MM-DD HH:MM -->`。

#### F. 继续下一个任务

输出简短过渡，自动进入下一个任务：
```
✅ 任务 X.Y 完成 → 继续任务 X.Z...
```

---

### 第三步：阻塞处理

遇到以下情况时暂停并请求决策：

| 阻塞类型 | 处理方式 |
|----------|----------|
| 任务描述不清晰 | 暂停，列出 2-3 种可能的理解，请用户选择 |
| 依赖的前置任务未完成 | 暂停，提示先完成哪个任务 |
| 涉及文件的当前代码与预期不符 | 展示差异，询问是跳过、手动修复还是按当前状态继续 |
| 需要外部信息（API key、数据库连接等） | 暂停，明确告知需要什么 |
| 多个合理实现方式 | 暂停，简要列出选项（2-3 个），推荐其一 |

**暂停消息模板：**
```
⏸️ 任务 [X.Y] 暂停 — [阻塞原因]

[详细说明问题和可能的解决路径]

A) [选项 1]
B) [选项 2]
C) 跳过此任务，稍后回头处理

你的选择？
```

---

### 第四步：阶段完成

每个阶段的所有任务完成后，输出阶段总结并自动进入下一阶段：
```
🏁 阶段 X 完成：[阶段名称]
   完成 N/N 任务
   
   产出：
   - [x] [文件 1]
   - [x] [文件 2]
   
   → 继续阶段 X+1...
```

阶段间不暂停，自动推进。

---

### 第五步：全计划完成

所有任务完成后，等待用户确认：
```
🎉 计划执行完毕！

总结：
- 新建文件：N 个
- 修改文件：M 个
- 完成阶段：X/X

请确认结果。你可以：
- 回复「确认完成」结束本次工作流
- 回复「检查 X」让我验证特定部分
- 回复「回退任务 X.Y」让我重新执行某个任务
```

**必须收到用户明确确认后才能视为流程结束。** 绝不未经确认自行结束。

---

## 执行规则

### 原子性原则

- 每个任务要么完全完成，要么清楚标记阻塞原因
- 不跳过依赖未满足的任务
- 任务失败不影响已完成的任务（不自动回滚）

### 进度可恢复

- 所有进度通过 PLAN.md 中的 checkbox 持久化
- 意外中断后重新加载 PLAN.md 即可从上次中断处继续
- 用户可以说"从任务 2.3 重新开始"来覆盖之前的部分执行

### 代码质量底线

即使按计划自动化执行，也必须保证：
- 代码可编译/可运行（不提交语法错误）
- 类型正确（TypeScript 项目必须通过类型检查）
- 不引入明显的安全漏洞（SQL 注入、XSS 等）

### 不要过度执行

- 严格控制在做任务描述范围内的事
- 发现规范或计划的合理性问题时，暂停并指出，而非自行修改范围
- 发现已有代码的问题时，记录但不顺手修改（除非在当前任务范围内）

### 与规范、计划技能的衔接

- 执行过程中如果频繁遇到需求不明确，应建议回到 `spec-clarify` 补充细节
- 如果发现任务拆分不合理（过大/过小/顺序不对），应建议回到 `plan-from-spec` 调整计划
