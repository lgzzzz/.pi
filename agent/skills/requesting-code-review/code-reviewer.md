# 代码评审要求

按照以下要求进行代码评审：

**目的：** 在工作量进一步累积之前，根据需求和代码质量标准对已完成的工作进行评审。

```
  提示词: |
    按照以下标准对已完成的代码变更进行评审，在问题产生连锁反应之前识别缺陷。评审应覆盖软件架构、设计模式、最佳实践等方面。

    ## 实施的内容

    {DESCRIPTION}

    ## 需求 / 计划

    {PLAN_OR_REQUIREMENTS}

    ## 待评审的 Git 范围

    **起始 (Base):** {BASE_SHA}
    **结束 (Head):** {HEAD_SHA}

    ```bash
    git diff --stat {BASE_SHA}..{HEAD_SHA}
    git diff {BASE_SHA}..{HEAD_SHA}
    ```

    ## 检查内容

    **计划一致性：**
    - 实施方案是否与计划/需求相匹配？
    - 偏差是合理的改进，还是存在问题的背离？
    - 计划的所有功能是否都已实现？

    **代码质量：**
    - 职责是否清晰分离？
    - 是否有恰当的错误处理？
    - 是否在适用的地方保证了类型安全？
    - 遵循 DRY 原则且没有过早抽象？
    - 是否处理了边缘情况？

    **架构：**
    - 设计决策是否合理？
    - 可扩展性和性能是否合理？
    - 是否存在安全隐患？
    - 是否与周围代码整洁集成？

    **测试：**
    - 测试是否验证了真实行为，而非仅模拟 (mocks) 行为？
    - 是否覆盖了边缘情况？
    - 在关键处是否有集成测试？
    - 所有测试是否通过？

    **生产就绪性：**
    - 如果架构改变，是否有迁移策略？
    - 是否考虑了向后兼容性？
    - 文档是否完整？
    - 是否存在明显的 bug？

    ## 校准标准

    按实际严重程度对问题进行分类。并非所有问题都是“严重 (Critical)”的。
    在列出问题之前，先确认做得好的地方 —— 准确的表扬有助于实施者信任后续的反馈。

    如果你发现与计划有重大偏差，请专门标出，以便实施者确认该偏差是否为刻意为之。
    如果你发现是计划本身而非实施方案存在问题，请指出。

    ## 输出格式

    ### 优点
    [哪些地方做得好？请具体说明。]

    ### 问题

    #### 严重 (必须修复)
    [Bug、安全问题、数据丢失风险、功能损坏]

    #### 重要 (应该修复)
    [架构问题、缺失功能、错误处理不当、测试缺漏]

    #### 次要 (锦上添花)
    [代码风格、优化机会、文档完善]

    针对每个问题：
    - 文件:行号 参考
    - 哪里错了
    - 为什么重要
    - 如何修复（如果不是显而易见的）

    ### 建议
    [针对代码质量、架构或流程的改进建议]

    ### 评估

    **是否准备好合并？** [是 | 否 | 修复后即可]

    **理由：** [1-2 句技术评估]

    ## 关键规则

    **要：**
    - 按实际严重程度分类
    - 具体（文件:行号，不要模糊）
    - 解释每个问题为什么重要
    - 肯定优点
    - 给出明确的结论

    **不要：**
    - 在未检查的情况下说“看起来不错”
    - 将琐碎的问题标为“严重”
    - 对你实际上没有阅读的代码提供反馈
    - 用词模糊（如“改进错误处理”）
    - 回避给出明确的结论
```

**占位符：**
- `{DESCRIPTION}` —— 对构建内容的简要摘要
- `{PLAN_OR_REQUIREMENTS}` —— 应该实现的功能（计划文件路径、任务文本或需求）
- `{BASE_SHA}` —— 起始提交
- `{HEAD_SHA}` —— 结束提交

**输出内容：** 优点、问题（严重 / 重要 / 次要）、建议、评估结果

## 输出示例

```
### 优点
- 具有恰当迁移的整洁数据库架构 (db.ts:15-42)
- 全面的测试覆盖（18 个测试，涵盖所有边缘情况）
- 带有回退机制的良好错误处理 (summarizer.ts:85-92)

### 问题

#### 重要
1. **CLI 包装器中缺失帮助文本**
   - 文件: index-conversations:1-31
   - 问题: 没有 --help 标志，用户无法发现 --concurrency 参数
   - 修复: 添加 --help 情况并附带使用示例

2. **缺少日期验证**
   - 文件: search.ts:25-27
   - 问题: 无效日期会静默返回空结果
   - 修复: 验证 ISO 格式，若不匹配则抛出错误并附带示例

#### 次要
1. **进度指示器**
   - 文件: indexer.ts:130
   - 问题: 对于耗时操作，缺少 "X of Y" 计数器
   - 影响: 用户不知道需要等待多久

### 建议
- 为了用户体验，建议添加进度报告
- 考虑为排除的项目设置配置文件（提高移植性）

### 评估

**是否准备好合并：修复后即可**

**理由：** 核心实施方案稳健，架构和测试良好。重要问题（帮助文本、日期验证）容易修复，且不影响核心功能。
```
