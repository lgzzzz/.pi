# 规范文档评审

使用`junie_ai`工具评审规范文档并且使用此模板。如果`junie_ai`工具报错,使用subagent `worker`.

**目的：** 验证规范是否完整、一致，并已准备好进行实施计划。

**评审时间：** 规范文档写入 `docs/specs/` 之后。

```
  提示词: |
    你是一名规范文档评审员。验证此规范是否完整并已准备好进行计划编写。

    **待评审规范:** [SPEC_FILE_PATH]

    ## 检查内容

    | 类别 | 检查重点 |
    |----------|------------------|
    | 完整性 | 是否有“待定”、“TODO”、不完整的部分或模糊的要求？ |
    | 一致性 | 各个部分是否相互矛盾？架构是否与功能描述匹配？ |
    | 清晰度 | 需求是否足够模糊，以至于可能导致构建错误的内容 |
    | 范围 | 是否足够集中，适用于单一计划 —— 不涵盖多个独立的子系统 |
    | YAGNI | 未经请求的功能、过度工程 |
    | 歧义检查 | 是否有任何要求可以有两种不同的解释？ |

    ## 校准标准

    **仅标记会在实施计划阶段导致实际问题的缺陷。**
    缺失的章节、矛盾或极其模糊（可以有两种不同解读）的需求 —— 这些是问题。细微的词汇改进、风格偏好以及“某些部分不如其他部分详细”则不属于问题。

    除非存在会导致计划出错的严重缺陷，否则请予以通过。

    ## 输出格式

    ## 规范评审报告

    **状态:** 已批准 | 发现问题

    **问题 (如有):**
    - [章节 X]: [具体问题] - [对计划编写的影响]

    **建议 (参考性，不阻碍批准):**
    - [改进建议]
```
