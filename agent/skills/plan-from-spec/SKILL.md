# 计划生成（Plan from Spec）

将规范文档中的需求转化为具体的、有序的、可执行的任务清单，使实现过程有据可依，降低遗漏和返工风险。

## 何时使用此技能

- 规范文档（SPEC.md 或 specs/ 下文档）已产出并确认
- 用户说"开始计划"、"拆分任务"、"接下来怎么做"
- 规范中存在多个相互依赖的子任务，需要理清顺序
- **用户对规范明确表示批准后**（如"批准""确认"）

**前置条件：** 存在已确认且用户批准的规范文档。如果规范不存在或未经批准，应先触发 `spec-clarify`。

**⚠️ 工作流门控规则：** 本技能完成后**必须等待用户明确批准**（如"批准""确认""执行"），才能进入 `execute-plan` 阶段。绝不未经批准自动推进。

---

## 工作流程

### 第一步：读取规范

从 SPEC.md（或用户指定的路径）读取完整规范。重点关注：

- 功能需求列表
- 数据模型与接口定义
- 交互流程
- 边界情况与错误处理
- 非功能性约束（性能、安全、兼容性等）

如有不明确之处，标注为「待确认」而非猜测填充。

---

### 第二步：任务拆分原则

按以下原则将规范拆分为任务：

1. **单一职责：** 每个任务只做一件事，描述清晰、边界明确
2. **可验证：** 每个任务有明确的完成标准（能跑、能测、能看效果）
3. **有序依赖：** 按依赖关系排序，先基础后上层
4. **粒度适中：** 每个任务 15-60 分钟可完成，避免过大或过小
5. **文件定位：** 明确每个任务涉及的文件和模块

**常见拆分维度：**

| 维度 | 说明 | 示例 |
|------|------|------|
| 数据层优先 | 先建模型/表/类型，再写逻辑 | 定义 schema → 实现 CRUD → 写 UI |
| 由底向上 | 先基础设施，后业务功能 | 工具函数 → 服务层 → 路由/组件 |
| 按模块切分 | 独立子系统各自成任务 | 认证模块 → 权限模块 → 通知模块 |
| 按用例切分 | 每个用户场景一个任务 | 注册流程 → 登录流程 → 找回密码 |
| 核心先行 | 先实现主流程，后补充边界 | 正常路径 → 错误处理 → 优化 |

---

### 第三步：任务清单模板

输出以下格式的 PLAN.md：

```markdown

**日期：** YYYY-MM-DD
**总任务数：** N
```

### 第四步：确认与调整

1. **展示计划**给用户，突出关键路径和风险项
2. **接受调整：** 用户可能想增减任务、调整顺序、合并拆分
3. **确认后保存**为项目根目录下的 `PLAN.md`（或 `plans/<名称>.md`）
4. **提示下一步（等待批准）：** "计划已就绪。请审阅任务拆分和顺序——这是执行的前提。批准后说「执行」或「开始」触发 `execute-plan`，也可以说「调整任务 X」让我修改计划。"未收到明确批准前，绝不自动进入执行阶段。

## 执行规则

### 任务粒度检查

如果拆分后的任务超过 15 个，考虑是否过于细碎；如果只有一个任务或每个任务预计超过 2 小时，考虑是否太粗。

### 隐式依赖识别

常见容易被忽略的依赖：
- 类型/接口定义必须在实现前完成
- 数据库迁移必须先跑
- 依赖包必须先安装
- 环境变量/配置必须先就位
- 测试数据必须先准备好

### 与执行技能的衔接

计划输出时提醒：
```
💡 计划已生成。你可以：
- 回复「执行」或「开始」让我按计划逐任务执行
- 回复具体任务编号（如「先做 2.3」）让我从指定任务开始
- 回复「跳过阶段 1」调整执行范围
```
