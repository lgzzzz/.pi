# 计划生成（Plan from Spec）

将规范文档中的需求转化为具体的、有序的、可执行的任务清单，使实现过程有据可依，降低遗漏和返工风险。

## 何时使用此技能

- 规范文档（SPEC.md 或 specs/ 下文档）已产出并确认
- 用户说"开始计划"、"拆分任务"、"接下来怎么做"
- 规范中存在多个相互依赖的子任务，需要理清顺序
- **用户对规范明确表示确认后**（如"确认"）

**前置条件：** 存在已确认且用户确认的规范文档。如果规范不存在或未经确认，应先触发 `spec-clarify`。

**⚠️ 工作流门控规则：** 本技能完成后**必须等待用户明确确认**（如"确认""执行"），才能进入 `execute-plan` 阶段。绝不未经确认自动推进。

---

## 工作流程

### 第零步：定位规范文件

如用户未指定规范路径，使用以下规则定位：
1. 在项目根目录的 `specs/` 目录下查找按 `YYYY-MM-DD_<slug>.SPEC.md` 格式命名的文件
2. 按日期降序排列，优先使用最新的规范文件
3. 若存在多个，列出最近 3 个供用户选择
4. 若不存在，提示用户先完成 `spec-clarify` 阶段

### 第一步：读取规范

从 SPEC.md（或用户指定的路径）读取完整规范。重点关注：

- 功能需求列表
- 数据模型与接口定义
- 交互流程
- 边界情况与错误处理
- 非功能性约束（性能、安全、兼容性等）

如有不明确之处，标注为「待确认」而非猜测填充。

---

### 第二步：任务拆分原则

按以下原则将规范拆分为任务：

1. **单一职责：** 每个任务只做一件事，描述清晰、边界明确
2. **可验证：** 每个任务有明确的完成标准（能跑、能测、能看效果）
3. **有序依赖：** 按依赖关系排序，先基础后上层
4. **粒度适中：** 每个任务 15-60 分钟可完成，避免过大或过小
5. **文件定位：** 明确每个任务涉及的文件和模块

**常见拆分维度：**

| 维度 | 说明 | 示例 |
|------|------|------|
| 数据层优先 | 先建模型/表/类型，再写逻辑 | 定义 schema → 实现 CRUD → 写 UI |
| 由底向上 | 先基础设施，后业务功能 | 工具函数 → 服务层 → 路由/组件 |
| 按模块切分 | 独立子系统各自成任务 | 认证模块 → 权限模块 → 通知模块 |
| 按用例切分 | 每个用户场景一个任务 | 注册流程 → 登录流程 → 找回密码 |
| 核心先行 | 先实现主流程，后补充边界 | 正常路径 → 错误处理 → 优化 |

---

### 第三步：任务清单与文件名

**文件名生成规则：** 基于规范的 slug 和当前日期，生成对应的 plan 文件名：
- 格式：`plans/YYYY-MM-DD_<slug>.PLAN.md`
- slug 与对应 SPEC 文件的 slug 保持一致
- 若 slug 对应的 SPEC 文件名为 `specs/2026-06-11_user-login.SPEC.md`，则 PLAN 文件名为 `plans/2026-06-11_user-login.PLAN.md`

输出以下格式的 PLAN.md：

```markdown
# 执行计划：[功能/变更名称]

**基于规范：** [SPEC 路径]
**文件：** plans/YYYY-MM-DD_<slug>.PLAN.md
**日期：** YYYY-MM-DD
**总任务数：** N
**预计总耗时：** X-X 小时

---
```

### 第四步：子代理评审（循环，直到无问题）

计划文档生成后，**自动触发子代理评审**，确保计划可行且完整。评审为独立循环：**子代理返回问题 → 主代理修复 → 重新评审 → 直到子代理不返回任何问题，即为通过。**

1. **发送评审：** 使用 `delegate` 工具，指定 `agent` 为 `plan-reviewer`，`task` 中仅包含计划文件路径和评审请求。**绝不**在 task 中提及这是第几轮评审、之前有过哪些反馈、或任何历史评审记录。
2. **等待结果：** 子代理返回评审意见
3. **判断结果：**
   - **无任何问题（通过）：** 进入用户确认环节（第五步），循环结束
   - **有问题：** 主代理根据反馈逐一修复计划文档中的问题，保存后回到第 1 步，重新启动子代理评审
4. **重复上述循环**，直到子代理返回“无问题”为止

**循环规则：**
- 最多 **7 轮**。超过 7 轮仍未通过，将剩余未解决问题列出，交给用户决策
- **每轮完全独立：** 每次调用子代理时，它收到的只是一个普通的评审任务——一份计划文件路径 + 一句「请评审这份计划」。子代理不知道这是第几轮、不继承上一轮的反馈记忆、不感知之前的修改历史
- **所有修复由主代理完成：** 子代理只负责发现问题，不参与修复；主代理根据反馈修改计划并保存，下一轮把修改后的计划当作全新文档发给子代理

### 第五步：确认与调整

1. **展示计划**给用户，突出关键路径、风险项及评审结果
2. **接受调整：** 用户可能想增减任务、调整顺序、合并拆分
3. **确认后保存**为 `plans/YYYY-MM-DD_<slug>.PLAN.md`（slug 与对应 SPEC 保持一致）
4. **提示下一步（等待确认）：** "✅ 计划已通过子代理评审，保存为 `plans/YYYY-MM-DD_<slug>.PLAN.md`。请审阅任务拆分和顺序——这是执行的前提。确认后说「执行」或「开始」触发 `execute-plan`，也可以说「调整任务 X」让我修改计划。"未收到明确确认前，绝不自动进入执行阶段。

## 执行规则

### 任务粒度检查

如果拆分后的任务超过 15 个，考虑是否过于细碎；如果只有一个任务或每个任务预计超过 2 小时，考虑是否太粗。

### 隐式依赖识别

常见容易被忽略的依赖：
- 类型/接口定义必须在实现前完成
- 数据库迁移必须先跑
- 依赖包必须先安装
- 环境变量/配置必须先就位
- 测试数据必须先准备好

### 与执行技能的衔接

计划输出时提醒：
```
💡 计划已生成。你可以：
- 回复「执行」或「开始」让我按计划逐任务执行
- 回复具体任务编号（如「先做 2.3」）让我从指定任务开始
- 回复「跳过阶段 1」调整执行范围
```
