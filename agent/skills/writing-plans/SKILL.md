# 编写计划 (Writing Plans)

## 概述

编写全面的实施计划。假设工程师对我们的代码库完全不了解且品味欠佳。记录他们需要知道的一切：每个任务要触及哪些文件、代码、测试、他们可能需要检查的文档、如何进行测试。将整个计划作为细粒度的任务交给他们。遵循 DRY、YAGNI、TDD 原则。频繁提交。

**启动时宣布：** “我正在使用 `writing-plans` 技能来创建实施计划。”

**计划保存至：** `docs/plans/YYYY-MM-DD-<feature-name>.md`
- （用户对计划位置的偏好优于此默认设置）

## 范围检查

如果规范涵盖了多个独立的子系统，则应在头脑风暴期间将其拆分为子项目规范。如果当时没有拆分，请建议将此计划拆分为多个独立的计划 —— 每个子系统一个。每个计划都应能独立产生可运行、可测试的软件。

## 文件结构

在定义任务之前，勾勒出将要创建或修改哪些文件，以及每个文件负责什么。这是锁定分解决策的地方。

- 设计具有清晰边界和定义良好的接口的单元。每个文件应有一个明确的职责。
- 当你可以同时将代码保存在上下文中时，你的推理效果最好；当文件重点明确时，你的编辑也更可靠。优先选择较小的、专注的文件，而不是承担过多任务的大型文件。
- 经常一起更改的文件应放在一起。按职责拆分，而不是按技术层级拆分。
- 在现有代码库中，遵循既定的模式。如果代码库使用大型文件，不要单方面重构 —— 但如果你修改的文件已经变得臃肿，在计划中包含拆分步骤是合理的。

这种结构为任务分解提供了依据。每个任务都应产生独立且有意义的更改。

## 细粒度的任务划分

**每一步都是一个动作（2-5 分钟）：**
- “编写失败的测试” —— 步骤
- “运行它以确保它失败” —— 步骤
- “实现最少代码使测试通过” —— 步骤
- “运行测试并确保它们通过” —— 步骤
- “提交” —— 步骤

## 计划文档页眉

**每个计划都必须以该页眉开始：**

```markdown
# [功能名称] 实施计划

**目标：** [用一句话描述这要构建什么]

**架构：** [说明实现方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N: [组件名称]

**文件：**
- 创建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py:123-145`
- 测试: `tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试以验证其失败**

运行: `pytest tests/path/test.py::test_name -v`
预期: 失败 (FAIL)，提示 "function not defined"

- [ ] **步骤 3：编写最简实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试以验证其通过**

运行: `pytest tests/path/test.py::test_name -v`
预期: 通过 (PASS)

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 禁止占位符

每一步都必须包含工程师需要的实际内容。以下内容被视为**计划失败** —— 绝不要写它们：
- “待定”、“TODO”、“稍后实现”、“填入细节”
- “添加适当的错误处理” / “添加验证” / “处理边缘情况”
- “为上述内容编写测试”（没有实际测试代码）
- “类似于任务 N”（重复代码 —— 工程师可能会不按顺序阅读任务）
- 仅描述做什么而不展示如何做的步骤（代码步骤需要代码块）
- 引用了未在任何任务中定义的类型、函数或方法

## 记住
- 始终使用准确的文件路径
- 每一步都包含完整的代码 —— 如果某一步更改了代码，请展示代码
- 带有预期输出的准确命令
- 遵循 DRY、YAGNI、TDD，频繁提交

## 自检

- 在编写完完整计划后，审视计划(参考`plan-document-reviewer-prompt.md`)。
- 执行这个循环:如果评审计划发现有任何问题,修复计划,然后重新评审计划,直到没有任何问题。


## 执行交接

计划已完成并保存至 `docs/plans/<文件名>.md`。
